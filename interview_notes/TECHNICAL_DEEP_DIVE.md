## Prediction Pipeline

The project does not use RAG. There is no retrieval-augmented generation or LLM retrieval pipeline here. The pipeline is a classical ML prediction pipeline plus rule-based recommendation logic.

The pipeline is:

```text
recent machine cycles
  -> feature engineering
  -> trained regression model
  -> predicted RUL
  -> risk level
  -> maintenance recommendation
```

### Step 1: Recent Sensor Cycles

The model expects recent cycles for one machine. Each row includes:

- cycle number
- operating settings
- sensor values from `sensor_1` through `sensor_21`

The dashboard can use historical sample data or simulated recent cycles.

### Step 2: Feature Engineering

For inference, the system sorts cycles by cycle number and keeps the latest window. The default window size is saved with the model artifact.

For each sensor, it computes:

- latest value
- mean
- standard deviation
- minimum
- maximum
- simple trend

The trend is:

```text
(last_value - first_value) / number_of_steps
```

This gives the model both current state and recent degradation direction.

### Step 3: Model Prediction

The model is a scikit-learn pipeline:

- `StandardScaler`
- `RandomForestRegressor`

The model predicts Remaining Useful Life. Negative predictions are clamped to zero because negative RUL is not operationally meaningful.

### Step 4: Risk Mapping

Predicted RUL is mapped to uppercase risk levels:

```text
CRITICAL: RUL <= 10
HIGH:     RUL <= 30
MEDIUM:   RUL <= 80
LOW:      RUL > 80
```

This is intentionally simple and explainable.

### Step 5: Maintenance Recommendation

Recommendations are rule-based, not generated by an LLM. Each risk level has a practical recommendation:

- `LOW`: continue normal operation
- `MEDIUM`: plan maintenance in the next service window
- `HIGH`: schedule immediate inspection
- `CRITICAL`: stop non-essential operation and dispatch maintenance

## Key Algorithms Or Logic

### RUL Target Creation

For each machine unit, RUL is calculated from the final observed cycle:

```text
RUL = max_cycle_for_unit - current_cycle
```

This converts time-series operating history into supervised training labels.

### Window-Based Feature Engineering

The model does not train directly on raw sequences. It converts recent time windows into tabular features. This keeps the implementation understandable and compatible with scikit-learn.

Tradeoff:

- Good for a portfolio project because it is explainable and easy to serve.
- Less powerful than sequence models for complex temporal behavior.

### Random Forest Regression

Random Forest was chosen because it is:

- strong as a baseline
- robust for tabular features
- easy to train locally
- easy to persist with `joblib`
- simple to explain in interviews

Tradeoff:

- It is not necessarily the best model for long sequence dynamics.
- It does not provide calibrated uncertainty by default.
- It can overfit if validation is not carefully designed.

## Important Engineering Decisions

### Shared Predictor Service

The best engineering decision is keeping prediction logic in `MaintenancePredictor`.

Why it matters:

- FastAPI and Streamlit direct mode reuse the same inference code.
- The deployment mode changes only the route to prediction.
- It reduces risk of inconsistent local vs cloud predictions.

### Dual Prediction Modes

`api` mode demonstrates backend integration:

```text
Streamlit -> FastAPI -> Predictor -> SQLite
```

`direct` mode supports Streamlit Cloud:

```text
Streamlit -> Predictor
```

This is a practical deployment tradeoff. The project still shows backend skills locally, while the public demo remains deployable for free.

### Deployment Artifact Fallback

Streamlit Cloud can start without local generated artifacts. The dashboard handles missing or empty demo data/model files by preparing them at runtime for the default demo path.

This solved a real deployment issue where empty placeholder files existed but were unusable.

### Import Path Fix

Because Streamlit Cloud runs `app/dashboard.py` directly, the repository root may not be on Python's import path. The dashboard adds the project root to `sys.path` before local imports.

This fixed:

```text
ModuleNotFoundError: No module named 'src'
```

### Dashboard UI

The dashboard is built to look like an industrial monitoring system:

- dark theme
- high-contrast cards
- risk badge
- health status cards
- trend charts
- recommendation panel

The goal is to communicate model output as an operational decision.

## Limitations

- The included data is synthetic/sample data, not a real production asset dataset.
- The model is a baseline Random Forest, not a fully optimized predictive maintenance model.
- Validation uses a simple split and should be improved for real machine-level generalization.
- SQLite is local and not suitable for high-concurrency production use.
- Direct Streamlit deployment does not write new prediction history.
- Risk thresholds are fixed rules, not learned or calibrated from business cost.
- There is no authentication, monitoring, CI/CD, or model registry.
- The system does not use RAG or LLM-generated recommendations.

## How I Would Improve This In Production

- Use real equipment data with clearly documented failure labels.
- Split train/test by machine unit or time period to better measure generalization.
- Compare Random Forest against gradient boosting, survival models, and sequence models.
- Add uncertainty estimates or prediction intervals.
- Calibrate risk thresholds with business cost and maintenance team input.
- Add feature importance or SHAP explanations.
- Deploy FastAPI separately and keep Streamlit as the frontend.
- Add model registry, version tracking, and rollback.
- Store predictions and sensor windows in PostgreSQL.
- Add CI/CD, unit tests, integration tests, and deployment checks.
- Add observability: request logs, model latency, failure rate, and drift metrics.
